# The Chronicles of QR Thrive: An API Journey

Welcome to the technical heart of QR Thrive. This document tells the story of how data flows through our system, from the moment a user first steps into our world to the detailed insights generated by a single scan.

---

## Chapter 1: The Gateway (Authentication)
*Every journey begins with an identity. In this chapter, we meet our user, Alex, who needs to securely enter the realm of QR Thrive.*

### 1.1 The Ritual of Joining (Signup)
Alex provides their name and choosing a secret key (password) to create an account.
- **Endpoint**: `POST /auth/signup`
- **Payload**:
```typescript
interface SignupDto {
  email: string;
  password: string;
  confirmPassword: string;
  firstName: string;
  lastName: string;
}
```
- **Response**: A User object and secure session cookies (`accessToken`, `refreshToken`).

### 1.2 The Return (Login)
Alex returns to the realm, proving their identity.
- **Endpoint**: `POST /auth/login`
- **Payload**:
```typescript
interface LoginDto {
  email: string;
  password: string;
  rememberMe?: boolean;
}
```
- **Response**: The authenticated User object and fresh session cookies.

### 1.3 The Mirror (Current User)
Alex checks their own reflection to ensure they are still recognized.
- **Endpoint**: `GET /auth/me` (Protected)
- **Response**: `{ user: { id: string, email: string, firstName: string, lastName: string } }`

## Chapter 2: Organizing the Realm (Folders)
*Alex is a tidy person. Before creating magic, Alex sets up "Boxes" (Folders) to keep their creations organized.*

### 2.1 Building a Box
Alex creates a folder named "Marketing 2026" with a vibrant blue color.
- **Endpoint**: `POST /folders` (Protected)
- **Payload**:
```typescript
interface CreateFolderDto {
  name: string;
  color: string; // Hex code
}
```

### 2.2 Surveying the Boxes
Alex looks at all their folders to see how many items are inside each.
- **Endpoint**: `GET /folders` (Protected)
- **Response**: `Array<BackendFolder>`
```typescript
interface BackendFolder {
  id: string;
  name: string;
  color: string;
  _count: { qrCodes: number };
}
```

## Chapter 3: The Art of Creation (QR Codes)
*Now, the magic begins. Alex designs a digital gateway—a QR Code.*

### 3.1 Forging a New Code
Alex creates a "Summer Sale" QR code. It's a dynamic URL type, styled with rounded dots and a custom logo.
- **Endpoint**: `POST /qr-codes` (Protected)
- **Payload**:
```typescript
interface CreateQRCodeDto {
  name: string;
  description?: string;
  folderId?: string;
  type: QRType;
  isDynamic?: boolean;
  data: any;   // Destination info (URL, vCard, etc.)
  design: any; // UI options (dots, corners, background)
  frame: any;  // Frame style and text
  logo?: string;
  width?: number;
  height?: number;
  margin?: number;
}
```

### 3.2 The Enums of Power
The system recognizes specific types of magic:
```typescript
enum QRType {
  url, text, vcard, wifi, email, sms, whatsapp, phone,
  instagram, facebook, linkedin, twitter, youtube, tiktok,
  crypto, socials, image, event, pdf, video, mp3, app,
  business, menu
}

enum QRStatus {
  active, archived
}
```

---

### 3.3 Searching the Collection
Alex wants to find all "Active" codes that match the word "Sale".
- **Endpoint**: `GET /qr-codes` (Protected)
- **Filters (Query Params)**:
  - `status`: 'active' or 'archived'
  - `folderId`: Filter by specific folder
  - `type`: Filter by QRType
  - `search`: Search in name or description

## Chapter 4: The Interaction (Public Scans)
*An adventurer (a customer) finds Alex's QR code in the wild and scans it with their phone.*

### 4.1 The Invisible Scribe (The Redirect)
The scan hits our system. Before the adventurer even sees the destination, our system records their presence.
- **Endpoint**: `GET /qr-codes/scan/:shortId` (Public)
- **Function**: 
  1. Identifies the user's device (Mobile/Desktop) via User-Agent.
  2. Identifies the user's location (City/Country) via IP address.
  3. Increments the scan count.
  4. Redirects the adventurer to Alex's destination.

## Chapter 5: The Oracle (Analytics)
*Alex returns to the dashboard to see the fruits of their labor. The system tells Alex exactly who came and from where.*

### 5.1 The High-Level Vision
Alex looks at the big picture: total scans, unique visitors, and device trends.
- **Endpoint**: `GET /qr-codes/stats` (Protected)
- **Response Data**:
```typescript
interface DashboardStats {
  totalQRs: number;
  totalScans: number;
  uniqueVisitors: number;
  deviceDist: Record<string, number>;  // e.g., { mobile: 42, desktop: 10 }
  osDist: Record<string, number>;      // e.g., { iOS: 30, Android: 12 }
  browserDist: Record<string, number>; // e.g., { Chrome: 20, Safari: 22 }
  chartData: Array<{ name: string, scans: number, unique: number }>; // Last 7 days
}
```

### 5.2 Examining a Single Artifact
Alex zooms in on one specific QR code to see its last 10 scans.
- **Endpoint**: `GET /qr-codes/:id` (Protected)
- **Response**: Includes the QR configuration and a `scans` array containing detailed location and device data for recent visitors.

## Chapter 6: The Evolution (Lifecycle)
*Nothing is static. Alex improves their work.*

### 6.1 The Mirror Image (Duplicate)
Alex likes the "Summer Sale" design so much they want to use it for "Autumn Sale".
- **Endpoint**: `POST /qr-codes/:id/duplicate` (Protected)
- **Function**: Clones the design and data into a new record with a new `shortId`.

### 6.2 The Archive
The sale is over. Alex hides the code from the active list without deleting the data.
- **Endpoint**: `PUT /qr-codes/:id` (Protected)
- **Payload**: `{ status: 'archived' }`

### 6.3 Finality
A code is no longer needed. Alex removes it forever.
- **Endpoint**: `DELETE /qr-codes/:id` (Protected)
